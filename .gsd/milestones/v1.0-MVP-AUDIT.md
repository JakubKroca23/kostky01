## Concerns
- **Verification Rigor**: Phase 6 was marked complete in the roadmap despite scoring discrepancies that were only resolved during the final milestone check.
- **State Volatility**: Current game state is entirely in-memory on the server. A server restart will wipe all active games and scores.
- **Code Parity**: `scoring.js` logic was manually duplicated between client and server. Future changes must be carefully synchronized to avoid "ghost points" or validation errors.

## Recommendations
1. **DRY Scoring**: In a future milestone, consider moving common logic into a shared package or better-automated sync script.
2. **Persistence**: Implement basic room persistence (e.g., Redis or a simple file-based cache) to survive server reboots.
3. **Automated Audits**: Add the scoring test to a CI/CD-like workflow to prevent regression in rules.

## Technical Debt to Address
- [ ] Implement Redis or database for room/player persistence (State safety).
- [ ] Refactor common utils (scoring, audio constants) into a shared structure if project grows.
- [ ] Add more comprehensive error handling for socket disconnects/timeouts in `App.jsx`.
